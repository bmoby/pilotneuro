## 1. Комментарии (АБСОЛЮТНЫЙ ПРИОРИТЕТ)

1. **Каждый файл `.tsx` начинается с заголовочного комментария в строке 0**, который простым языком объясняет:

   - для чего нужен этот файл,
   - что он показывает на экране,
   - что человек может с этим сделать.

   Пример оформления:

```tsx
/* Этот файл определяет страницу профиля.
    Он показывает данные вошедшего пользователя (имя, фото, email).
    Он также позволяет менять некоторые простые детали, например город или имя.
   */
```

2. **У каждого важного элемента должен быть комментарий выше:**

- элемента, который хранит информацию для страницы,
- элемента, который срабатывает автоматически при загрузке,
- функций, которые запускаются по клику или отправке формы,
- крупных частей интерфейса (список, форма, модальное окно и т. д.).

Комментарии пишутся простым языком, без жаргона (без «hook», «state», «props» и т. п.).
Если читать только комментарии в файле, человек без опыта в коде должен понять, что делает страница. 3. **Комментарии всегда актуальны:**

- если поведение меняется, комментарий сразу меняется,
- не должно оставаться ложных, расплывчатых или устаревших комментариев.

## 2. Структура файлов и папка `components`

1. В проекте **всегда есть папка `components**`.
2. **Все переиспользуемые компоненты** (кнопки, карточки, секции, формы, визуальные блоки и т. д.) создаются в `components/`.
3. Файлы страниц (в `app/` или `pages/`) должны оставаться максимально простыми:

- они используют компоненты из папки `components`,
- они не дублируют большие куски интерфейса.

## 3. Запрещено

Во всём проекте:

1. **Никаких HOC (Higher-Order Components)**

- Никогда не создавать компонент, который принимает компонент на вход и возвращает новый компонент.

2. **Никаких render props**

- Никогда не передавать функцию, предназначенную для «отрисовки» содержимого, как основную prop.

3. **Никаких кастомных хуков**

- Никогда не создавать функции, которые начинаются с `use`, чтобы инкапсулировать логику.
- Вся логика остаётся внутри самих компонентов.

## 4. Стили и CSS (Строгая изоляция)

1. **Принцип «Один компонент — один файл стилей»:**

- Любой визуальный компонент или страница должны иметь свой собственный, отдельный файл CSS.
- Мы **никогда не смешиваем** стили разных компонентов в одном файле.

2. **Приоритет структуры над сокращением кода:**

- Даже если это создает дублирование кода, каждый компонент должен быть изолирован.
- Мы предпочитаем когерентность структуры файлов и полную изоляцию логики/стилей друг от друга.
